## 实验结论

在本机 CPU-only、同一 GGUF 权重、同一提示词、同一 batch/ubatch、同一线程数的条件下：

- `llama.cpp llama-server` 的顺序单请求吞吐为 `94.288` completion tok/s。
- 魔改 Nano vLLM 的 `model_backend=gguf` 吞吐为 `85.006` completion tok/s。
- llama.cpp 在这组服务级 CPU 测试中约快 `1.11x`，Nano 平均端到端延迟高约 `10.9%`。
- 成对并发请求时，两边都会因为共同抢 CPU 降速；llama.cpp 为 `75.352` tok/s，Nano GGUF 为 `70.859` tok/s，差距约 `1.06x`。

这不是 GPU/native CUDA benchmark；整个测试只覆盖 CPU-only GGUF 推理。

## 瓶颈分析

1. 两套系统底层都在跑 GGUF CPU 推理。llama.cpp 走原生 `llama-server.exe`，Nano vLLM 走新增 `model_backend=gguf`，内部通过 `llama-cpp-python` 调 llama.cpp。
2. Nano GGUF 比 llama.cpp 慢约 10% 左右，主要差异更可能来自 Python/FastAPI/adapter 层、请求对象构造、结果封装和 backend 串行锁，而不是模型权重或量化格式差异。
3. 成对并发时两边都下降，说明瓶颈是 CPU decode 资源竞争。这个阶段用于模拟两个服务同时驻留，不适合作为单系统最佳吞吐排名。
4. llama-server 日志显示它会维护 slot/context 状态；因为用户要求同样提示词，多轮测试会体现服务端重复请求路径。报告保留这一点作为 caveat，而不是把它解释成 GPU 或其他环境优势。
5. 如果下一步继续 CPU-only 优化，建议只做线程数 sweep：`threads=1/2/4/8/16`，固定模型、prompt、batch、ubatch，找本机 CPU 的吞吐拐点。不要把当前结果外推到 CUDA native scheduler。
